## 1. 产品概述

### 1.1 产品定位
《与哲对话》是一款以"哲学家人格化 AI 对话"为核心的知识陪伴类产品。用户可以在哲学家图鉴中浏览平台收录的哲学家资料（如尼采、苏格拉底、孔子等），并选择任意一位哲学家进行沉浸式对话，体验其思想方式、语言风格与价值观视角，从而获得知识科普 + 情感陪伴 + 思辨启发的复合价值。

### 1.2 核心价值主张
- **知识可对话化**：把静态的哲学知识（生平、思想、著作）转化为可交互的对话体验，降低哲学学习门槛。
- **人格化 AI**：每位哲学家拥有独立的人设 Prompt、语言风格与价值立场，而非千人一面的通用助手。
- **多端触达**：H5（社交传播/免安装）、Web（PC 深度使用）、小程序（微信生态低门槛获客）三端覆盖不同场景。

### 1.3 目标用户
- 对哲学、心理学、人文社科感兴趣的泛知识用户（学生、白领）
- 希望通过"角色扮演式对话"获得思辨启发或情绪疏导的用户
- 内容/教育行业从业者，用作科普或教学辅助工具

### 1.4 MVP 范围界定【假设，需确认】
本期（v1.0）仅聚焦两大核心界面，暂不包含社区、UGC 共创哲学家、付费订阅等能力，具体见第 4 章。

## 2. 产品范围与三端说明

| 端 | 定位 | 技术形态建议 | 关键差异点 |

### 4.2 哲学家图鉴模块

#### 4.2.1 哲学家列表页
- **展示形式**：卡片列表，含头像/画像、姓名、所属流派标签（如"存在主义""德国古典哲学"）、一句话标签（如尼采："上帝已死，超人哲学"）。
- **排序/筛选**：默认推荐排序；支持按流派、时代（古希腊/近代/现代）、地域（西方/东方）筛选【假设，需确认收录范围是否含中国哲学家如孔子、老子】。
- **搜索**：支持按姓名/关键词模糊搜索。
- **交互**：点击卡片进入哲学家详情页。

#### 4.2.2 哲学家详情页
- **基础信息区**：姓名（含外文原名）、生卒年、国籍/所属流派、头像/画像。
- **思想简介区**：核心思想概述（结构化短文，非大段堆砌，建议 300-500 字 + 3-5 个关键词标签，如"权力意志""永恒轮回""超人"）。
- **代表著作**：书目列表，可选是否附简介。
- **对话入口**：醒目的"开始与 XX 对话"按钮，点击后：
  - 若该哲学家已有历史会话 → 弹出选择"继续上次对话"或"发起新对话"
  - 若无历史会话 → 直接进入聊天框，AI 以该哲学家人设发送开场白

#### 4.2.3 哲学家数据结构【建议字段，供后端设计参考】
```json
{
  "philosopher_id": "nietzsche",
  "name": "弗里德里希·尼采",
  "name_en": "Friedrich Nietzsche",
  "avatar_url": "",
  "birth_year": 1844,
  "death_year": 1900,
  "school": ["存在主义", "德国古典哲学"],
  "era": "近代",
  "region": "西方",
  "tagline": "上帝已死，超人哲学",
  "bio": "……",
  "key_concepts": ["权力意志", "永恒轮回", "超人", "上帝已死"],
  "representative_works": [
    {"title": "查拉图斯特拉如是说", "year": 1883, "intro": "……"}
  ],
  "persona_prompt": "（系统级，不直接展示给用户，见 5.1）",
  "opening_line": "你相信自己是自己命运的主人吗？"
}
```

### 4.3 聊天框模块

#### 4.3.1 会话列表页
- **展示内容**：每条会话展示对方哲学家头像+姓名、最近一条消息摘要、最后更新时间。
- **排序**：按最后消息时间倒序。
- **操作**：点击进入会话；长按/滑动可删除、置顶【假设】。
- **入口**：底部导航栏 Tab（图鉴 / 会话列表 / 我的），或图鉴页对话后自动跳转。

#### 4.3.2 聊天详情页（对话界面）
- **消息展示**：
  - 用户消息右侧气泡，AI（哲学家）消息左侧气泡，附哲学家头像。
  - AI 回复采用流式（打字机）效果输出，提升"正在思考/表达"的拟人感。
  - 支持消息内 Markdown 简单渲染（分段、强调）。
- **输入区**：文本输入框、发送按钮；【假设】暂不支持语音输入（可作为 P2 后续迭代）。
- **顶部信息栏**：当前对话的哲学家头像+姓名，点击可跳转回其详情页；提供"结束/新建对话"入口。
- **上下文管理**：单一会话内维持多轮对话记忆，超出模型上下文窗口时做滑动窗口裁剪或摘要压缩（见第 5 章）。
- **异常状态**：AI 响应超时/失败时的重试提示；网络异常提示。

## 5. AI 对话系统设计

### 5.1 哲学家人设 Prompt 设计原则
每位哲学家对应一份独立的系统级人设 Prompt（不对用户暴露），建议包含以下要素：
1. **身份锚定**：明确"你是尼采本人，以第一人称与用户对话"，而非"你是一个了解尼采的助手"。
2. **语言风格**：句式特征、口吻（如尼采式的格言体、诘问式表达）。
3. **知识边界**：其思想体系内的核心概念、代表著作、历史背景，避免时代错位的现代知识断言【重要：需明确 AI 应如何处理"你怎么看待互联网/AI"这类跨时代提问——建议引导角色以其思想框架做类比回应，而非生硬拒答】。
4. **对话姿态**：是否主动追问、是否具有一定"挑战/反问用户"的思辨风格。
5. **安全边界**：禁止输出医疗、法律、心理危机干预等专业建议；涉及自伤/自杀等敏感信息时触发统一的安全兜底话术并引导用户寻求专业帮助（与哲学家人设弱耦合，此规则应作为系统层硬约束，优先级高于人设扮演）。

### 5.2 开场白机制
用户首次与某哲学家发起对话时，由该哲学家资料中预设的 `opening_line` 触发首条 AI 消息，降低"面对空白输入框"的启动门槛。

### 5.3 上下文与记忆策略【技术方案建议，需研发评估】
- 短期：单会话内滑动窗口保留最近 N 轮对话。
- 中期（P1+）：会话摘要机制，超长对话自动生成摘要注入 system 层，保留话题连贯性。
- 是否跨会话记忆用户偏好（如"用户是学生"）：本期【假设：不做】，作为后续千人千面方向。

### 5.4 内容安全
- 所有哲学家人设 Prompt 输出需经过统一的内容安全审核层（涉政、涉黄、暴力等违规内容过滤），尤其小程序端需满足微信平台内容合规要求。
- AI 生成内容需在界面适当位置提示"内容由 AI 生成，观点不代表平台立场"【小程序合规通常要求此类免责声明】。

## 6. 用户体系与账号

【假设，需产品侧确认】
- 支持游客模式直接体验（限制：不保留历史会话，或仅保留本地 24 小时）。
- 登录方式：Web/H5 支持手机号或第三方登录；小程序支持微信一键登录，实现同一用户在小程序端与 Web/H5 端数据是否互通需确认（涉及统一账号体系设计）。
- 登录后可长期保存会话列表、跨设备同步。

## 7. 非功能性需求

| 类别 | 要求 |
|---|---|
| 性能 | AI 首字响应时间（TTFB）建议 ≤ 2s，流式输出保持稳定节奏 |
| 并发 | 需评估 AI 接口调用成本与并发限流策略，避免高峰期超时 |
| 兼容性 | H5 需适配主流移动浏览器与微信内置浏览器；小程序遵循微信小程序开发规范 |
| 合规 | 涉及 AI 生成内容的产品需符合《生成式人工智能服务管理暂行办法》等相关规定，做好内容标识与审核 |
| 数据安全 | 用户对话内容加密存储，明确隐私政策与数据留存策略 |
| 可扩展性 | 哲学家数据结构、人设 Prompt 管理建议后台可配置化，便于持续新增哲学家而无需改代码 |

## 8. 技术架构建议（供 Cursor 立项参考）

### 8.1 整体架构
```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│   Web 端     │  │   H5 端      │  │  小程序端    │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │
       └────────────────┼────────────────┘
                         │ HTTPS / WebSocket(流式)
                  ┌──────▼──────┐
                  │  API 网关/BFF │
                  └──────┬──────┘
        ┌────────────────┼────────────────┐
   ┌────▼────┐     ┌─────▼─────┐    ┌─────▼─────┐
   │ 用户/会话 │     │ 哲学家配置  │    │  AI 对话   │
   │  服务     │     │   服务     │    │  编排服务   │
   └────┬────┘     └─────┬─────┘    └─────┬─────┘
        │                │                 │
   ┌────▼────┐     ┌─────▼─────┐    ┌─────▼─────┐
   │ 关系型DB  │     │  配置DB/CMS│    │ LLM API    │
   │(用户/会话)│     │(哲学家数据) │    │(Claude/等) │
   └─────────┘     └───────────┘    └───────────┘
```

### 8.2 技术选型建议【供参考，非强制】
- **跨端方案**：若希望一套代码覆盖三端，可评估 uni-app / Taro；若追求各端体验最优，建议 Web 用 React/Vue + H5 响应式共用一套代码，小程序原生或 Taro 单独开发。
- **后端**：Node.js/Python 均可，重点是 AI 编排层（Prompt 组装、流式转发、上下文管理）需独立服务化，便于后续切换/多模型接入。
- **流式输出**：Web/H5 可用 SSE 或 WebSocket；小程序端需注意微信小程序对流式请求的兼容方式（分包接收或轮询兜底）。
- **哲学家配置**：建议做成后台可配置的 CMS（人设 Prompt、头像、简介等），避免每新增一位哲学家都要发版。

### 8.3 数据模型（核心表）【建议】
- `philosophers`：哲学家基础信息表（见 4.2.3）
- `users`：用户表
- `conversations`：会话表（user_id, philosopher_id, created_at, updated_at, title）
- `messages`：消息表（conversation_id, role, content, created_at）
